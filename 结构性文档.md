# 蓝图编排者：系统级个人旅行导演 Agent

## 业务流程与页面结构文档 V1.0

项目基于 AIGC 应用赛道，核心是把大模型能力落到 vivo 终端设备与系统能力中，形成“从意图理解到跨端执行”的完整 AI 出行体验。比赛通知中也明确提到应用赛道关注移动终端 AI 应用开发与 vivo 终端落地能力。 你现有方案中的“时空拓扑看板、跨平台决策中枢、全模态意图注入、呼吸感重排”等方向已经在原作品稿中形成雏形。

---

## 1. 产品一句话

**蓝图编排者 Blue-Map Orchestrator** 是一个基于 BlueLM 与 vivo OS 系统能力的个人旅行导演 Agent。
它能从语音、文本、PDF、截图、订单通知中理解用户意图，自动生成弹性时空拓扑行程，并通过地图、日历、闹钟、天气、出行 App、酒店餐饮平台完成跨端执行。

---

## 2. 核心用户场景

用户输入：

> “下周五去北京出差三天，住西单附近，想吃正宗烤鸭，周六下午想去故宫。”

系统自动完成：

1. 解析目的地、日期、偏好、约束。
2. 读取日历，避开已有会议。
3. 查询天气、交通、人流。
4. 生成可视化时空拓扑行程。
5. 区分硬锚点和软任务。
6. 用户可拖动节点、追加需求、上传改签截图。
7. 系统自动重排行程。
8. 点击节点可直接跳转携程、美团、高德、日历、闹钟等应用。

# 3. 总业务流程

## 阶段一：意图爆发期 Input & Perception

### 用户行为

用户通过任意方式发起计划：

文本输入、语音输入、上传文件、上传图片、快捷标签补充。

### 输入类型

| 输入类型 | 示例             | 系统处理              |
| ---- | -------------- | ----------------- |
| 长文本  | “下周五去北京出差三天……” | BlueLM 语义解析       |
| 语音   | 口述完整需求         | 语音转文本 + 意图解析      |
| PDF  | 出差通知、会议通知      | 文档解析 + 关键信息提取     |
| 图片   | 机票截图、酒店截图、攻略截图 | Vision OCR + 信息识别 |
| 快捷标签 | 出发地、目的地、日期、预算  | 半结构化补全            |

### 核心后端逻辑

BlueLM 提取五要素：

| 要素            | 内容               |
| ------------- | ---------------- |
| Action 动作     | 出差、住宿、用餐、参观、交通   |
| Location 地点   | 北京、西单、故宫、会议中心    |
| Time 时间       | 下周五、三天两晚、周六下午    |
| Constraint 约束 | 航班时间、会议时间、酒店位置   |
| Preference 偏好 | 正宗烤鸭、少走路、预算、景点类型 |

### 调用系统能力

| 能力      | 用途         |
| ------- | ---------- |
| 日历 API  | 检查会议冲突     |
| 天气 API  | 查询目的地未来天气  |
| 地图 API  | 路线、距离、交通方式 |
| 路况 API  | 拥堵预测       |
| POI API | 酒店、餐厅、景点   |
| Vision  | 识别票据、截图、通知 |
| 通知识别    | 捕捉订单变更     |

## 阶段二：视觉转译期 Visual Synthesis

### 目标

把语言计划转化为一个“可看、可点、可缩放、可执行”的时空拓扑看板。

### 主要对象

| 对象                 | 含义         | UI 表达        |
| ------------------ | ---------- | ------------ |
| 硬锚点 Hard Anchor    | 不可随意改动的节点  | 实体卡片 + 锁图标   |
| 软任务 Soft Task      | 可调整的活动     | 半透明气泡        |
| 动态连线 Dynamic Link  | 节点之间的路径    | 发光线条 + 交通图标  |
| 风险提示 Risk Layer    | 时间紧张、天气、拥堵 | 轻量红线/橙色提示    |
| 语义缩放 Semantic Zoom | 宏观/微观切换    | 城市骨干网 / 区域细节 |

### 节点分级

| 类型   | 示例            | 是否可拖动 |
| ---- | ------------- | ----- |
| 硬锚点  | 航班、高铁、会议、酒店入住 | 不可拖动  |
| 半硬锚点 | 已预订餐厅、预约景点    | 可建议修改 |
| 软任务  | 吃饭、逛街、买特产     | 可拖动   |
| 推荐节点 | 系统建议的空白填充     | 待用户确认 |

## 阶段三：动态微调期 Interactive Refining

### 目标

用户不用重新输入完整需求，只在已有拓扑上增量修改。

### 输入方式

| 输入方式  | 示例              |
| ----- | --------------- |
| 拖动节点  | 把“故宫”拖到下午       |
| 语音追加  | “会议结束后帮我加一个买特产” |
| 文本追加  | “把午饭换成便宜一点的”    |
| 上传新截图 | 改签短信、酒店变更       |
| 批量选择  | 同时调整几个软任务       |

### 核心交互机制

| 机制     | 说明           |
| ------ | ------------ |
| 锚点锁定   | 航班、会议不可随意移动  |
| 弹性拉伸   | 软任务随硬锚点自动重排  |
| 冲突红染   | 时间冲突时只染红相关路径 |
| 空白填充   | 识别空档，推荐附近活动  |
| B 计划生成 | 冲突严重时给出替代方案  |

### 示例流程

用户说：

> “周五下午会议提前结束了，帮我在回酒店路上加一个买特产的地方，要步行。”

系统执行：

1. 解析“会议提前结束”为硬约束变更。
2. 更新会议节点结束时间。
3. 查找会议中心到酒店之间的步行可达 POI。
4. 插入“买特产”软任务节点。
5. 重算路线时间。
6. 若不影响晚餐和酒店入住，自动加入。
7. 若时间紧张，显示轻量风险提示。

## 阶段四：跨端执行期 Action & Linkage

### 目标

Agent 从“规划导演”变成“执行场控”。

不是简单打开 App，而是把前面解析出的参数带入第三方平台。

### 参数化跳转

| 节点 | 跳转 App       | 传入参数           |
| -- | ------------ | -------------- |
| 酒店 | 携程 / 飞猪 / 美团 | 城市、日期、区域、房型、预算 |
| 餐饮 | 美团 / 大众点评    | 菜系、距离、营业时间、评分  |
| 交通 | 高德 / 百度地图    | 起点、终点、方式、出发时间  |
| 机票 | 携程 / 航旅纵横    | 航班号、日期、机场      |
| 会议 | 日历           | 时间、地点、提醒       |
| 天气 | 系统天气         | 城市、小时级天气       |
| 闹钟 | 系统闹钟         | 出发提醒、签到提醒      |

### URI / Deep Link 示例

```text
ctrip://hotel/search?city=北京&checkin=2026-05-10&checkout=2026-05-12&keyword=西单&room=大床房

meituan://poi/search?keyword=北京烤鸭&location=西单&time=13:00

amapuri://route/plan/?sourceApplication=BlueMap&dname=故宫博物院&dev=0&t=2

calendar://create?title=互联网大会&start=2026-05-10T10:45&end=2026-05-10T12:00
```

# 11. Agent 设计框架（工程实现版 V1.1）

## 11.1 总体架构

```text
移动端 D1~D5
    ↓ HTTP
FastAPI 编排层
    ↓
LangGraph Agent 工作流（可降级为顺序执行）
    ├─ Intent Engine（DashScope qwen-plus 结构化抽取）
    ├─ Context Hub（日历 / 天气 / POI / 路况 mock）
    ├─ Plan Generator（规则引擎 + 模板行程）
    ├─ Refining Engine（指令微调）
    ├─ Guardian（异常检测 + 重规划）
    └─ Linkage Executor（deeplink / 系统同步）
```

## 11.2 阶段映射（文档 ↔ App ↔ API）

| 文档阶段 | App 页面 | 关键 API | Agent 节点 |
| --- | --- | --- | --- |
| 意图爆发期 | D1 需求输入 + D1 解析确认 | `/intent/analyze` `/voice/transcribe` | `parse_intent` |
| 视觉转译期 | D2 方案比对 + 时空拓扑 | `/plans/compare` | `generate_plan` |
| 动态微调期 | D4 动态守护 | `/guardian/replan` | `refiner` |
| 跨端执行期 | D3 确认订票 | `/orders/*` `/sync/system` | `execution_center` |
| 回顾沉淀 | D5 回顾沉淀 | `/trips/{id}/review` | `trip_review` |

## 11.3 LangGraph 状态机

```text
parse_intent
  → retrieve_context（RAG 附件检索）
  → call_tools（天气 / 日历冲突 / POI）
  → generate_plan（行程节点生成）
  → validate_plan（冲突 / 密度校验）
  → persist_plan（SQLite 持久化）
  → respond（用户可见回复）
```

**降级策略：** Python 3.13 未安装 LangGraph 时，按同样顺序同步执行，保证演示可用。

## 11.4 五要素抽取协议

| 要素 | 字段来源 | 示例 |
| --- | --- | --- |
| Action | `intent.preferences` + 规则 | 出差 / 住宿 / 餐饮 / 游览 |
| Location | `origin/destination/must_visit` | 北京 / 西单 / 故宫 |
| Time | `start_date/end_date/duration_days` | 下周五 / 3天2晚 |
| Constraint | `constraints` | 住西单附近 |
| Preference | `preferences` | 烤鸭 / 少走路 |

移动端在 **D1 解析确认页** 展示五要素 + 系统上下文（日历无冲突、目的地天气、路况预警）。
